## 📌 今日核心收获

Context 不是"给模型塞更多资料"，而是一套**信息治理问题**——你要为每类信息标注来源、时效、权限和可信度。真正难的不是塞内容，而是把系统规则、用户目标、历史状态、工具结果和外部文档分清楚。

---

## 🧠 知识节点

### 1. Context Window（上下文窗口）

模型一次请求能处理的最大上下文范围。窗口越大，能放进来的资料越多，但**不代表模型会完美使用每个细节**。

**长上下文的陷阱**：「看见了但没抓住重点」。如果把合约源码、审计报告、治理讨论和交易日志全部塞进去，模型可能：
- 被无关内容分散注意力
- 引用过期段落

**最佳实践**：context window 应该配合**检索、摘要和结构化数据**使用：
- 关键状态直接给
- 长文档按需取
- 低可信内容隔离标注

---

### 2. Context Engineering（上下文工程）

设计上下文**进入模型的方式**——包括选择数据源、排序、裁剪、标注来源、隔离不可信内容、决定哪些信息必须每次重新查询。

**一个稳定的工具型 Agent 上下文**，不只是"用户问题 + 一段 JSON"，还应包括：

| 层级 | 内容 |
|------|------|
| 任务状态 | 当前任务进度 |
| 工具返回 | 工具调用结果 |
| 日志/证据 | 相关日志或验证记录 |
| 可信来源 | 可信数据源 |
| 外部检查 | 外部校验结果 |
| 用户意图 | 用户原始意图 |
| 约束 | 系统禁止事项 + 输出 schema |

**目标不是塞满窗口，而是让模型在正确的信息层级里工作。**

---

### 3. Memory（记忆）

跨请求保留的信息：用户偏好、历史任务、常用钱包、项目配置、上次分析结果。

⚠️ **隐藏风险**：
- 记住"用户常用 0xabc 地址" → 下次默认它是收款地址
- 记住"用户风险偏好较高" → 放松高风险交易确认

🔑 **核心原则**：**Memory 不能替代实时授权。** 用户过去允许某个动作，不代表现在仍然允许。所有涉及身份、权限、资产或外部副作用的记忆，都必须重新绑定当前会话和当前授权。

---

### 4. Knowledge Base（知识库）

系统可检索的外部知识库：产品文档、SDK 文档、代码说明、论坛讨论、FAQ、内部 runbook。

⚠️ **知识库 ≠ 自动正确**：如果不维护来源、更新时间、版本和废弃状态，RAG 只会把**旧错误更稳定地送进模型**。

**知识库至少应记录**：
- 文档来源和 URL
- 最后更新时间
- 适用协议版本
- 适用版本或运行环境
- 是否来自官方或第三方
- 是否需要人工复核

## 🔗 在 AI × Web3 中的位置

Context 是连接模型和现实系统的入口。AI 负责推理，Web3 负责状态和结算；**context 决定模型看到的是用户幻想、过期文档，还是可验证的链上事实。**

一个靠谱的 Agent 上下文分为五层：

```
指令层  → 系统规则、工具使用规则、禁止事项
任务层  → 用户目标、本次会话参数
事实层  → 链上状态、工具结果、simulation
知识层  → 文档、标准、论坛、历史案例
记忆层  → 用户偏好和项目配置
```

**层级越清楚，后续越容易做权限控制、prompt injection 防护和审计。**

## 🛠️ 最小实践（今日作业）

### 任务：为"钱包授权检查 Agent"设计 context spec

**场景**：用户问「这个 dApp 要我 approve，可以签吗？」

**模型回答前必须拿到的上下文**：

| 字段 | 来源 | 时效要求 |
|------|------|----------|
| chain id + 当前区块 | 实时查询 | 必须实时 |
| token 合约、spender 地址、approve 数量 | 从交易解析 | 必须实时 |
| 用户当前 allowance + 余额 | 链上查询 | 必须实时 |
| spender 是否在可信列表 | 可信列表缓存 | 可缓存（含更新时间） |
| simulation 或静态检查结果 | 实时执行 | 必须实时 |
| dApp 页面说明 | 外部内容 | 标记为**不可信** |
| 用户本次意图 | 用户输入 | 当前会话 |

**需要明确标注的规则**：
- dApp 页面说明 → **不可信外部内容**，模型中隔离
- spender 可信列表 → 可缓存但需标注最后更新时间
- 链上状态 → 必须实时查询，不可复用上次结果
- 用户意图 → 每次重新确认，不用 memory 推断

## 🤔 个人思考

1. **Context 是安全的第一道防线**：之前学 Prompt Injection 时关注点在 prompt 本身，现在意识到 context 的**分层设计**才是防注入的核心——把不可信内容隔离在独立层级，不让它混入系统指令。

2. **Memory 的边界感**：以前觉得 AI 越能"记住我"越好，现在明白好的 memory 设计要像**权限系统**——记住偏好但不清除每次重新授权，记住配置但不自动绑定身份。

3. **知识库的维护成本被低估了**：建一个 RAG 很容易，但维护文档版本、标注废弃信息、区分官方/第三方来源，才是决定 RAG 是"帮手"还是"帮倒忙"的关键。

4. **五层分层法很实用**：指令/任务/事实/知识/记忆的分层思路不只适用于 Web3 Agent，连我现在用的 Hermes Agent 也是类似的设计——系统 prompt（指令层）+ 当前消息（任务层）+ 工具结果（事实层）+ skills（知识层）+ memory（记忆层）。
