## 📝 学习笔记

### 为什么要学 LLM

很多 AI 应用一开始都会说"我们用大模型"。这句话还不够。需要继续追问：
- 模型负责理解用户意图，还是生成内容？
- 它只是在回答问题，还是能调用工具？
- 它的错误会停留在文本里，还是会进入真实工作流？

**LLM 是今天大多数 AI 应用的能力底座**，能：
- 总结文档
- 写代码
- 生成计划
- 抽取结构化信息
- 把自然语言转成程序格式

**但它本身不是数据库**，不会自动知道外部世界的最新状态，也不能替系统承担事实校验。

### 第一性原理

> LLM 的核心能力来自**概率建模**：给定前文，预测下一个最可能的 token。

这决定了 LLM 在任何严肃系统里的位置：
- ✅ 可以帮你理解、归纳、生成和规划
- ❌ 不能单独承担事实来源、权限判断和最终执行

**越靠近真实动作，越需要**：
- 外部数据
- 确定性规则
- 人工或系统校验

### 知识节点

#### 1. Token

Token 是模型处理文本的**基本单位**，不一定等于一个汉字或单词，而是 tokenizer 切分后的片段。

**Token 直接影响三件事**：
1. **上下文能放多少** - 长文档、代码、JSON 容易塞满
2. **调用成本是多少** - 按 token 计费
3. **模型能不能完整看见关键信息**

**💡 实践提示**：
- 代码、JSON、长标识符比普通段落更吃 token
- 需要决定哪些信息原样放入，哪些先压缩，哪些交给检索系统按需取回

#### 2. Embedding

Embedding 是把文本、代码或其他对象**映射成向量**，用来衡量"语义上是否接近"。

**常用于**：
- 搜索
- 聚类
- 推荐
- 异常检测
- RAG（检索增强生成）

**⚠️ 限制**：
- 向量相似度只能说明内容接近
- **不能替代来源校验、规则检查和人工判断**

#### 3. Transformer

Transformer 是现代 LLM 的**核心架构**，关键能力来自 **attention（注意力机制）**：
- 模型可以在生成时关注输入中的不同位置
- 学习词、代码、事实和上下文之间的关系

**工程事实**：
- Transformer 擅长在上下文里找模式
- **不等于拥有稳定数据库**
- 可能因为上下文缺失或相似模式误导而给出错误归纳

#### 4. Hallucination（幻觉）

Hallucination 指模型生成了**看起来合理、但并不真实或无法验证的内容**。

**常见表现**：
- 编造 API
- 错误解释代码
- 引用不存在的资料
- 把旧版本文档当成当前状态

**风险等级**：
- 普通问答：质量问题
- 带执行能力的系统：**流程风险**（错误参数、错误权限解释、错误操作建议都可能进入自动化链路）

### 在 AI x Web3 中的位置

LLM 在 AI x Web3 系统里通常承担：
1. **意图理解** - 把用户自然语言转成结构化目标
2. **内容生成** - 生成解释、总结、代码片段
3. **模式匹配** - 从链上数据、文档、日志中找规律

**但永远需要外部校验**：
- 链上状态 ≠ 模型记忆
- 合约逻辑 ≠ 模型推理
- 资产操作 ≠ 文本生成

### 最小实践

**目标**：用 API 调用一个 LLM，观察 token 消耗、响应时间和输出质量。

**步骤**：
1. 选一个模型 API（OpenAI、Claude、本地模型）
2. 准备一段混合中英文的输入文本
3. 记录：
   - 输入 token 数
   - 输出 token 数
   - 响应时间
   - 输出质量评估
4. 尝试让模型解释一段链上交易数据，观察它是否能正确理解
