## 📝 学习笔记

### 为什么要学 Context

LLM 没有自动连接你的数据库、文件系统、API、用户历史和项目文档。它只会基于当前上下文生成输出。如果上下文错了、缺了、过期了，模型再强也会给出不可靠答案。

很多 AI 应用的问题表面看是"模型不够聪明"，实际是 context 设计混乱：
- 把不可信网页当系统指令
- 把旧文档当最新规则
- 把用户愿望当事实
- 把上一次任务的状态泄漏到下一次任务

在任何带工具调用的场景里，这都会直接影响安全。模型如果看不到真实工具结果，就只能猜；Agent 如果不知道当前状态、权限和预算，就可能生成越界操作。

### 第一性原理

> Context 不是简单的长文本拼接，而是一套**信息治理问题**。你要为每类信息标注来源、时效、权限和可信度。否则模型会把"用户说的""网页写的""链上查到的""系统规定的"混在同一层处理。

核心原则：
- **来源标注**: 每条信息都要标明从哪来的
- **时效管理**: 过期信息要有明确标记
- **权限隔离**: 不同权限级别的信息分开处理
- **可信度分层**: 可信、不可信、待验证信息分层标注

### 知识节点

#### 1. Context Window（上下文窗口）

- 模型一次请求能处理的最大上下文范围
- **窗口越大 ≠ 用得越好**: 长上下文常见问题是"看见了但没抓住重点"
- 如果把合约源码、审计报告、治理讨论和交易日志全部塞进去，模型可能被无关内容分散注意力，也可能引用过期段落
- **最佳实践**: context window 应该配合检索、摘要和结构化数据使用。关键状态直接给，长文档按需取，低可信内容隔离标注

#### 2. Context Engineering（上下文工程）

设计上下文进入模型的方式，包括：
- 选择数据源
- 排序与裁剪
- 标注来源
- 隔离不可信内容
- 决定哪些信息必须每次重新查询

一个稳定的工具型 Agent 上下文通常包括：
1. **系统指令**（角色、约束、安全规则）
2. **当前任务状态**（进行到哪一步，下一步做什么）
3. **可用工具列表**（每个工具的参数、返回值、使用条件）
4. **用户输入**（当前请求，标注来源和可信度）
5. **历史交互摘要**（关键决策，而非完整对话）
6. **实时查询结果**（链上数据、API 返回，带时间戳）
7. **Memory 摘要**（用户偏好、常用配置，但需定期确认）

**相关 topic**: Chain-aware Context, RAG

#### 3. Memory（记忆）

- 跨请求保留的信息：用户偏好、历史任务、常用钱包、项目配置、上次分析结果
- **风险**: Memory 可以让 Agent 更顺手，但也容易带来隐性风险
  - 系统记住"用户常用 0xabc 地址"，下次就默认它是收款地址
  - 记住"用户风险偏好较高"，就放松了高风险交易确认
- **原则**: Memory 不能替代实时授权。用户过去允许某个动作，不代表现在仍然允许

#### 4. Knowledge Base（知识库）

- 系统可检索的外部知识库：产品文档、SDK 文档、代码说明、论坛讨论、FAQ、内部 runbook
- **注意**: Knowledge Base 不是让模型"记住"所有内容，而是在需要时检索最相关的部分
- 需要配合 RAG 使用，而不是直接塞进 prompt

### 在 AI x Web3 中的位置

Context 是 AI 和 Web3 的**关键桥梁**:

| 场景 | Context 作用 |
|------|-------------|
| 链上数据分析 | 提供实时链状态、合约 ABI、用户持仓 |
| 交易助手 | 当前余额、Gas 价格、滑点设置、权限范围 |
| 智能合约审计 | 源码、依赖库、已知漏洞模式、审计历史 |
| DAO 治理 | 提案内容、投票历史、委托人偏好、规则约束 |

**Chain-aware Context** 是 AI × Web3 的核心概念：Agent 必须知道自己在哪条链上、当前区块高度、交易确认状态、gas 成本等链上上下文。

### 最小实践

**目标**: 设计一个"链上交易分析 Agent"的上下文结构。

```json
{
  "system": {
    "role": "链上交易风险分析助手",
    "constraints": ["不泄露私钥", "不执行未授权操作", "标记所有假设"]
  },
  "task_state": {
    "current_step": "分析待签名交易",
    "next_action": "输出风险报告"

  "user_input": {
    "query": "帮我看看这笔交易",
    "source": "user_chat",
    "trust_level": "high"
  },

    "chain_id": 1,
    "block_number": 19876543,
    "gas_price": "25 gwei",
    "native_balance": "1.5 ETH"
  },
  "transaction": {
    "to": "0x...",
    "value": "0 ETH",
    "data": "0x...",
    "estimated_gas": 150000
  },
  "memory": {
    "user_preference": "保守型（需人工确认 > 0.5 ETH）",
    "last_confirmed": "2026-05-20T10:00:00Z"
  }
}
```

## 💭 思考与 Feedback

### 对 Handbook 的反馈

1. **Context Window**: 希望增加不同模型的窗口大小对比表（GPT-4、Claude、Gemini 等）
2. **Context Engineering**: 可以提供更多实际产品的 context 设计案例
3. **Memory 安全**: 建议补充 Memory 的过期机制和确认流程设计

### 个人思考

- Context 设计是 AI 产品的隐形架构，用户看不见但决定了可靠性
- 信息治理 > 信息堆积，分层标注比简单拼接更重要
- Web3 场景对 Context 要求更高，因为涉及资产安全
- Chain-aware Context 是 AI × Web3 的核心竞争力
- Memory 是双刃剑，需要谨慎设计权限边界
